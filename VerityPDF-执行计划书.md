## 1. 项目概述

### 1.1 项目背景

在数字化转型浪潮中，PDF 文档作为信息交换的标准格式，被广泛应用于合同签署、技术文档分发、学术论文审阅、设计稿反馈等场景。尽管市场上存在 Adobe Acrobat、Foxit Reader 等商业产品，但它们存在以下痛点：

1. **高昂的授权费用**：Adobe Acrobat Pro 年订阅费用超过 ¥2,000/人，对企业团队来说成本高昂
2. **臃肿的体积**：完整安装包超过 1GB，启动时间长，资源占用大
3. **平台支持不均**：Linux 平台支持薄弱，跨平台体验不一致
4. **批注功能封闭**：标注数据格式私有，难以与其他工具集成
5. **扩展性差**：缺乏插件机制，无法根据工作流定制

VerityPDF 项目正是在此背景下启动，旨在利用开源技术栈（Electron + PDF.js + Konva.js）构建一款**轻量、免费、跨平台、可扩展**的专业 PDF 批注工具，填补开源社区在这一领域的空白。

### 1.2 项目愿景与使命

**愿景声明**：

> 打造一款开源、免费、专业级的 PDF 批注工具，在功能上对标 Adobe Acrobat 的标注模块，在性能上超越同类 Electron PDF 应用，成为开发者和知识工作者的首选 PDF 审阅工具。

**使命声明**：

> 通过开源协作，为每一位用户提供高质量、零门槛的 PDF 文档审阅体验，让文档批注变得简单、高效、可追溯。

**核心价值主张**：

| 价值维度 | 说明 |
|----------|------|
| **零成本** | MIT 开源协议，完全免费使用，无功能限制 |
| **高性能** | 亚秒级启动，流畅翻页，低内存占用 |
| **跨平台** | macOS / Windows / Linux 原生体验一致 |
| **隐私安全** | 本地运行，无遥测数据收集，支持离线使用 |
| **可扩展** | 模块化架构，支持插件和自定义工具 |
| **数据开放** | 标注数据使用开放 JSON 格式，不锁定用户 |

### 4.4 数据流架构

#### 4.4.1 PDF 文档加载流程

```
用户选择文件
    │
    ▼
[Main Process] ── fs.readFile ──► ArrayBuffer
    │
    ▼ IPC: 'pdf:load'
[Renderer] ── PDFService.loadDocument()
    │
    ├──► pdfjsLib.getDocument({ data, password? })
    │       │
    │       ├──► onProgress ──► pdfStore.setLoadingProgress()
    │       └──► Promise<PDFDocumentProxy>
    │
    ├──► 提取元数据 ──► pdfStore.setMetadata()
    ├──► 解析大纲 ──► pdfStore.setOutline()
    ├──► 生成页面列表 ──► pdfStore.setPages()
    └──► 渲染首页 ──► PDFPageRenderer.renderPage(1)
                        │
                        ├──► page.getViewport({ scale })
                        ├──► canvas.getContext('2d')
                        ├──► page.render({ canvasContext, viewport })
                        └──► 插入 DOM
```

#### 4.4.2 标注创建流程

```
用户选择工具（如矩形）
    │
    ▼
toolStore.setActiveTool('rect')
    │
    ▼
DrawingController.startDrawing()
    │
    ├──► mousedown ──► 记录起点坐标
    ├──► mousemove ──► 实时绘制预览矩形（Konva.Rect）
    └──► mouseup ──► 完成绘制
                        │
                        ▼
               CoordinateConverter.screenToPdf()
                        │
                        ▼
               AnnotationManager.addAnnotation({
                 type: 'rect',
                 position: { x, y },
                 size: { width, height },
                 style: { color, strokeWidth, opacity }
               })
                        │
                        ▼
               annotationStore.addAnnotation()
                        │
                        ├──► HistoryManager.push(action)
                        ├──► Konva Stage 更新
                        └──► 触发自动保存
```

#### 4.4.3 标注导出流程

```
用户点击"导出 PDF"
    │
    ▼
[Renderer] ── IPC: 'export:merge' ──► [Main Process]
                                        │
                                        ├──► pdf-lib: PDFDocument.load(originalPdf)
                                        ├──► 遍历标注列表
                                        │     │
                                        │     ├──► Rect ──► pdfDoc.getPage().drawRectangle()
                                        │     ├──► Text ──► pdfDoc.getPage().drawText()
                                        │     ├──► Line ──► pdfDoc.getPage().drawLine()
                                        │     ├──► Image ──► pdfDoc.getPage().drawImage()
                                        │     └──► ... (其他标注类型)
                                        │
                                        ├──► CoordinateConverter.pdfToExport()
                                        │     (确保坐标映射正确)
                                        │
                                        ├──► pdfDoc.save()
                                        └──► fs.writeFile(outputPath)
```

#### 4.4.4 自动保存流程

```
标注变更事件
    │
    ▼
useAutoSave Hook ── 防抖 30s（可配置）
    │
    ├──► 立即保存：Ctrl+S / Cmd+S
    └──► 定时保存：距上次变更 30s 后
              │
              ▼
       StorageService.saveProject()
              │
              ├──► 序列化 annotationStore.annotations
              ├──► 序列化 pdfStore.currentPage / zoom / rotation
              ├──► 构建 .verity JSON 结构
              │
              │   {
              │     "version": "1.0",
              │     "pdfPath": "/path/to/doc.pdf",
              │     "pdfHash": "sha256:...",
              │     "createdAt": "2026-06-15T...",
              │     "updatedAt": "2026-06-15T...",
              │     "annotations": [...],
              │     "viewState": { page: 1, zoom: 1.0 }
              │   }
              │
              ├──► 写入临时文件 .verity.tmp
              ├──► 原子替换 rename(.verity.tmp → .verity)
              └──► 更新 uiStore.saveStatus = 'saved'
```

### 14.2 风险应对策略详细方案

#### R01: PDF.js 渲染兼容性问题（极高风险）

| 维度 | 内容 |
|------|------|
| **触发条件** | Phase 2 测试中发现 > 1% 的 PDF 样本渲染异常 |
| **预防措施** | 1. 提前收集 50+ 种 PDF 测试样本（含 CJK/加密/大文件/扫描/图表）<br>2. 在 Sprint 3 初期进行 PDF.js POC 验证<br>3. 建立 PDF.js 渲染问题跟踪表 |
| **应急计划** | 1. 对已知问题文档显示"渲染不完全"提示<br>2. 维护 PDF.js fork 分支，自主修复关键 Bug<br>3. 考虑降级渲染模式（关闭部分特性） |
| **应急预案** | 如 PDF.js 无法满足需求，评估 MuPDF WASM 替代方案（预估 2-3 周工作量） |
| **监控指标** | 渲染正确率（目标 ≥ 99%），每周统计 |

#### R02: Konva.js 性能瓶颈（高风险）

| 维度 | 内容 |
|------|------|
| **触发条件** | 500 个标注对象同时显示时 FPS < 30 |
| **预防措施** | 1. 在 Sprint 5 初期进行 Konva.js 性能基准测试<br>2. 使用 Konva.Layer.batchDraw() 批量渲染<br>3. 标注 > 200 时启用简化渲染模式 |
| **应急计划** | 1. 实现可视区域标注裁剪（仅渲染可见标注）<br>2. 使用 OffscreenCanvas 后台渲染<br>3. 考虑降级为原生 Canvas 2D 直接绘制 |
| **应急预案** | 如 Konva.js 无法满足性能需求，评估 PixiJS 替代方案（预估 3-4 周） |
| **监控指标** | 标注渲染 FPS，每个 Sprint 测试 |

#### R03: 坐标系转换精度不足（高风险）

| 维度 | 内容 |
|------|------|

| **监控指标** | 坐标转换误差（目标 < 0.5px），每个 Sprint 验证 |

# GitHub Actions CI 流水线
name: CI

on:
  push:
    branches: [main, develop]
  pull_request:
    branches: [main]

jobs:
  lint:
    runs-on: ubuntu-latest

      - uses: actions/checkout@v4

    runs-on: ${{ matrix.os }}
    strategy:
      matrix:
        os: [macos-latest, windows-latest, ubuntu-latest]
    steps:
      - uses: actions/checkout@v4
      - run: npm ci
      - run: npm run test:unit -- --coverage
      - run: npm run test:integration

  build:
    runs-on: ${{ matrix.os }}
    needs: [lint, test]
    strategy:
      matrix:
        os: [macos-latest, windows-latest, ubuntu-latest]
    steps:
      - uses: actions/checkout@v4
      - run: npm ci
      - run: npm run build
      - uses: actions/upload-artifact@v4
        with:
          name: build-${{ matrix.os }}
          path: dist/

    runs-on: ${{ matrix.os }}
    needs: [build]
    strategy:
      matrix:
        os: [macos-latest, windows-latest, ubuntu-latest]

### 17.1 Git 分支策略

```
main (生产分支)
  │
  ├── release/v1.0 (发布分支)
  │     │
  │     ├── hotfix/xxx (紧急修复)
  │     │
  │     └── ... 
  │
  └── develop (开发分支)
        │
        ├── feature/pdf-renderer (功能分支)
        ├── feature/annotation-tools
        ├── feature/export-service
        ├── feature/i18n
        │
        ├── fix/xxx (修复分支)
        │
        └── ...
```

#### 分支命名规范

| 类型 | 命名格式 | 示例 |
|------|---------|------|
| 功能分支 | `feature/<功能描述>` | `feature/pdf-renderer` |
| 修复分支 | `fix/<issue号或描述>` | `fix/issue-42-memory-leak` |
| 热修复 | `hotfix/<描述>` | `hotfix/crash-on-export` |
| 发布分支 | `release/v<版本号>` | `release/v1.0` |
| 重构分支 | `refactor/<描述>` | `refactor/annotation-store` |

#### 合并策略

| 场景 | 策略 | 说明 |
|------|------|------|
| feature → develop | Squash Merge | 将多个 commit 压缩为一个 |
| fix → develop | Squash Merge | 保持 develop 历史整洁 |
| develop → release | Merge Commit | 保留完整的开发历史 |
| hotfix → main + develop | Cherry-pick | 同时修复到两个分支 |

### 17.2 版本管理策略

采用 [Semantic Versioning](https://semver.org/) 规范：

```
主版本号.次版本号.修订号-预发布标识

示例:
  1.0.0       → v1.0 GA 正式发布
  1.0.0-beta.1 → Beta 测试版本
  1.0.0-rc.1  → Release Candidate
  1.1.0       → 新增功能（向后兼容）
  1.0.1       → Bug 修复
  2.0.0       → 重大变更（不兼容）
```

### 17.3 变更控制流程

```
变更请求提交
    │
    ▼
变更影响评估（PM + TL）
    │
    ├── 影响小（< 2 人天）──► PM 审批 ──► 直接实施
    │
    ├── 影响中（2-5 人天）──► PM + TL 审批 ──► 纳入下一个 Sprint
    │
    └── 影响大（> 5 人天）──► 项目发起人审批 ──► 调整基线计划
                                                      │
                                                      ├── 调整范围（削减其他功能）
                                                      ├── 调整时间（延期发布）
                                                      └── 调整资源（增加人手）
```

## 附录 B：技术决策记录 (ADR)

### ADR-001：选择 Electron 作为桌面框架

- **日期**: 2026-05-15
- **状态**: 已采纳
- **背景**: 需要一个跨平台桌面框架，支持 macOS/Windows/Linux
- **决策**: 选择 Electron 30
- **理由**:
  1. 最成熟的跨平台桌面框架，社区庞大
  2. 基于 Chromium + Node.js，Web 技术栈完全可用
  3. PDF.js 和 Konva.js 在 Chromium 中表现最佳
  4. 丰富的 API（菜单、快捷键、文件对话框、自动更新）
- **后果**:
  - 安装包体积较大（基础 ~80MB），需优化
  - 内存占用较高，需精细管理
- **替代方案**: Tauri（Rust，体积小但生态弱）、NW.js（社区小）

### ADR-002：选择 PDF.js 作为渲染引擎

- **日期**: 2026-05-15
- **状态**: 已采纳
- **背景**: 需要一个高保真 PDF 渲染引擎
- **决策**: 选择 PDF.js 4.x
- **理由**:
  1. Mozilla 官方维护，持续更新
  2. 对 PDF 1.7 标准兼容度最高
  3. 支持 Web Worker 非阻塞渲染
  4. 开源 MIT 协议
- **后果**:
  - 部分特殊字体/加密文档可能渲染不完全
  - 需要维护 fork 分支以修复特定问题
- **替代方案**: MuPDF WASM（GPL 协议风险）、Poppler（C++ 绑定复杂）

### ADR-003：选择 Konva.js 作为标注引擎

- **日期**: 2026-05-16
- **状态**: 已采纳
- **背景**: 需要一个高性能 2D Canvas 交互库用于标注
- **决策**: 选择 Konva.js 9.x
- **理由**:
  1. 功能完善（Transformer、事件系统、图层管理）
  2. API 直观，开发效率高
  3. 支持批量渲染 (batchDraw)
  4. 社区活跃，文档丰富
- **后果**:
  - 大量标注 (>500) 时可能需要性能优化
  - 库体积约 150KB (gzipped)
- **替代方案**: Fabric.js（性能较弱）、Paper.js（API 不直观）、PixiJS（面向游戏，标注功能需自建）

### ADR-004：选择 Zustand 作为状态管理

- **日期**: 2026-05-16
- **状态**: 已采纳
- **背景**: 需要轻量级的状态管理方案
- **决策**: 选择 Zustand 4.x
- **理由**:
  1. 极轻量（< 1KB gzipped）
  2. API 简洁，无 boilerplate
  3. 支持 middleware（persist、devtools、immer）
  4. TypeScript 支持完善
- **后果**:
  - 社区生态不如 Redux 庞大
  - 大型应用状态拆分需要自行组织
- **替代方案**: Redux Toolkit（太重）、MobX（隐式响应式）、Jotai（原子化不适合集中状态）

### ADR-005：选择 Vite 作为构建工具

- **日期**: 2026-05-15
- **状态**: 已采纳
- **背景**: 需要快速的前端构建工具
- **决策**: 选择 Vite 5.x
- **理由**:
  1. 极速 HMR（毫秒级热更新）
  2. 开箱即用，配置简单
  3. 原生 ESM 支持
  4. 与 electron-vite 集成良好
- **后果**:
  - 部分 CommonJS 库可能需要额外配置
  - 生产构建使用 Rollup，部分 Webpack 插件不可用
- **替代方案**: Webpack（太慢）、esbuild（配置能力有限）

### ADR-006：.verity 项目文件格式

- **日期**: 2026-05-20
- **状态**: 已采纳
- **背景**: 需要一种方式持久化标注数据，且不修改原始 PDF
- **决策**: 使用 JSON 格式的 .verity 文件，与 PDF 同目录
- **理由**:
  1. 不修改原始 PDF 文件，保证数据完整性
  2. JSON 格式人类可读，易于调试
  3. 开放格式，便于第三方集成
  4. 文件名与 PDF 同名，自动关联
- **后果**:
  - 用户移动 PDF 时需同时移动 .verity 文件
  - 文件名冲突时需处理（同目录同名 PDF）
- **替代方案**: 嵌入 PDF 注释（修改原始文件）、SQLite 数据库（不可读、过度设计）

interface VerityAPI {
  // 文件操作
  openFile(): Promise<string | null>;
  saveFile(data: string, defaultPath: string): Promise<boolean>;
  readFile(filePath: string): Promise<ArrayBuffer>;

  // 应用信息
  getVersion(): string;
  getPlatform(): string;

  // 窗口控制
  minimizeWindow(): void;
  maximizeWindow(): void;
  closeWindow(): void;

  // 事件监听
  onMenuAction(callback: (action: string) => void): void;
  onFileOpen(callback: (filePath: string) => void): void;
  onBeforeClose(callback: () => Promise<boolean>): void;

  // 更新

}
